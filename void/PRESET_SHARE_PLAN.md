# Plan — поделиться пресетом по ссылке

> Этот файл — подробный план фичи «Share preset by URL» для проекта Void Typeface
> (`mishaivanov.ru/void`). Написан так, чтобы спустя год вернуться и продолжить
> реализацию без потери контекста. Если читаешь это впервые — пройдись по разделам
> по порядку: «Контекст» → «Что передавать» → «Ограничения» → «Решения по
> архитектуре» → «План этапов» → «Чего не делать» → «Открытые вопросы».

Дата создания: **2026-05-01** (Misha + Claude, перед началом другого спринта).

---

## 1. Контекст и цель

**Что хотим:**

1. Пользователь настраивает кастомный пресет.
2. Сохраняет его в локальной библиотеке.
3. В дропдауне рядом с пресетом появляется иконка «share» (только для
   пользовательских пресетов, не для встроенных seed-пресетов).
4. По клику в clipboard копируется ссылка вида
   `https://mishaivanov.ru/void/#p=<base64url payload>`.
5. Получатель открывает ссылку и видит ровно тот же пресет, что у автора
   (с тем же текстом, цветами, рандом-флагами и — при возможности —
   зафиксированными альт-глифами).

**Почему это в принципе делается:** в проекте уже сериализуемая модель
пресета (`PresetManager`, `presets/*.json`, `EXTRA_PRESET_SNAPSHOT_KEYS`).
Никакого бэкенда заводить не нужно — всё помещается в URL fragment.

---

## 2. Что именно входит в пресет (полезный объём данных)

В `localStorage` пресет хранится как обычный JS-объект, формируется методом
`PresetsController.collectPresetData()`. Состав:

- `settings.values` — ~80 полей: числа, булевы, hex-цвета, многострочный
  `text`. Типичный объём JSON: **0.8–2 KB**.
- `alternativeGlyphCache` — `{ "row_col": idxAlt }`. **Десятки байт.**
- `moduleTypeCache` — пер-буквенные оверрайды
  (`stem`, `strokesNum`, `strokeGapRatio`, `dashLength`, `gapLength`,
  `useDash`). Растёт с числом уникальных букв в `text`. **0.5–3 KB.**
- `moduleValueCache` — обычно пустой, но архитектурно поддержан.
- `colorPalette` — массив hex-цветов до ~32 шт. **150–400 байт.**
- `moduleColorCache` — `Map` по глобальному индексу модуля. В сильных Chaos
  пресетах с длинным текстом может быть **сотни записей → 5–15 KB**.

Все «не-`settings`»-поля уже описаны константой `EXTRA_PRESET_SNAPSHOT_KEYS`
в `js/controllers/PresetsController.js`. **Переиспользуем именно её**, не
изобретаем новый список.

## 3. Размер URL и почему это главный риск

- **Браузеры:** Chrome/Edge/Firefox без проблем держат 8–16 K в адресной
  строке, Safari — до ~80 K. На старте работы с проектом в 2026 это
  актуально, но проверять перед релизом.
- **Реальные мессенджеры (iMessage, Telegram, Slack, Discord)** не режут
  ссылки до нескольких килобайт.
- **Twitter/X / почтовые превью / QR** комфортны при **~1.5–2 K**.
- **Соцсети с агрессивными «трекерами»** иногда переписывают ссылки —
  fragment (`#…`) к ним не уходит, поэтому мы выбираем именно fragment.

**Сжатие** (`deflate` через `pako` или встроенный `CompressionStream` +
base64url-encode): на наших данных типично **×4–×8** ужатие — много
повторов в ключах JSON.

**Бюджет, к которому стремимся:**

| Сценарий                                  | Ожидаемая длина URL |
| ----------------------------------------- | ------------------- |
| Чистый пресет без рандома и без кэшей     | **<800 байт**       |
| Средний пресет с рандомом, без кэшей      | **1–2 K**           |
| Жирный Chaos-пресет с полными кэшами      | **3–6 K**           |

Порог, на котором показываем предупреждение / предлагаем «share without
exact randomization caches»: **~6 K**.

## 4. Решения по архитектуре

1. **Fragment, не query.** `#p=…` не уходит в server-логи и аналитику, не
   вызывает перезагрузок, не упирается в лимиты query-string на сервере
   (`414 URI Too Long`).
2. **Версионирование payload.** Префикс `v1.<base64>` (или поле `v: 1`
   внутри). У нас уже не раз мигрировала схема (`mode → dashEnabled`,
   `colorMode → colorSource + palette` и т.д. — см. `loadPreset`). Старые
   ссылки должны переживать миграции.
3. **Diff vs default.** Кодируем **только** поля, отличающиеся от
   пристинных defaults. У нас они уже снимаются в `main.js`
   (`pristineDefaults` перед инициализацией DOM). Это самая большая
   экономия — **уже снимает 60–70%** объёма.
4. **Чистка пустых полей.** `{}` / `[]` не пишем.
5. **Кэши делаются опциональными.** Без них рандомизированные пресеты
   дадут визуально другую картинку, но «такую же по духу». Для
   не-рандомных пресетов кэши не нужны вообще. Если включаем `includeCaches`
   — делаем по выбору пользователя (или автоматически, если payload
   укладывается в бюджет).
6. **`seeded: false` всегда** в шарингах. У получателя пресет ведёт себя
   как любой пользовательский, без обучающей анимации Random-панели и без
   привилегированного отношения.
7. **Применение на старте — без записи в `localStorage`.** Гость не должен
   засорять свою библиотеку. Делаем «временный текущий» пресет, и UI
   предлагает «Save to your library / Discard». После save — это уже его
   обычный пресет.
8. **Никакого бэкенда / shortener’ов.** Минусы (стоимость, зависимость,
   приватность, миграции БД) перевешивают плюсы для нашего масштаба.

## 5. Поэтапный план

### Этап 0. Аудит размеров (1 итерация)

Цель — откалибровать порог `~6 K` и принять решение «по умолчанию каши
включаем или нет».

- Снять реальный размер (raw JSON и compressed) для всех seed-пресетов
  из `presets/*.json` и для нескольких «синтетически жирных» — длинный
  текст + полные `paletteDice*` + рандом всех осей.
- Цифры записать прямо в этот файл (раздел 9 «Журнал замеров»).
- Решить: ставим ли `includeCaches` дефолтом, или показываем чекбокс.

### Этап 1. Кодек (модуль `js/share/PresetShareCodec.js`)

Чисто функциональный модуль, без зависимости на DOM:

- `encode(presetBlob, { includeCaches }) → string`:
  1. Diff с pristineDefaults (импортируется/прокидывается, не дублируется).
  2. Отброс пустых `{}` / `[]`.
  3. При `includeCaches=false` отрезать `moduleTypeCache`,
     `moduleValueCache`, `moduleColorCache`.
  4. Удалить `seeded`, `createdAt`, `updatedAt`, `id` (если такие есть).
  5. `JSON.stringify` → `pako.deflateRaw` (или `CompressionStream('deflate-raw')`).
  6. base64url (без `=` padding, символы `-_` вместо `+/`).
  7. Префикс `v1.` → `v1.<base64>`.
- `decode(string) → presetBlob` — обратная цепочка.
  - Если префикса нет — отказ.
  - Прогоняется через те же миграции, что и `loadPreset`. Удобный путь:
    добавить в `PresetManager` метод `normalizeForLoad(blob)`, в который
    вынести существующую цепочку миграций (mode, colorMode, paletteDice
    и т.п.), и из него же вызывать в `loadPreset`. Тогда декодер просто
    возвращает «нормализованный» blob, который равно подходит для
    `loadPreset`/`loadFromShareBlob`.
- Без зависимостей на UI, тестируемо в изоляции.

### Этап 2. UI: иконка «share» в дропдауне пресетов

Файл: `js/controllers/PresetsController.js`, метод `updatePresetList()`.

- Рядом с pencil-иконкой `.preset-dropdown-item-edit` добавить
  кнопку `.preset-dropdown-item-share` (тот же паттерн поведения).
- Показывать **только** если: `name !== 'New' && name !== 'Unsaved' && !preset.seeded`.
- На клик:
  1. `presetData = collectPresetDataFor(name)` (для конкретного пресета,
     не «текущего»; такой геттер удобно завести).
  2. `link = encode(presetData, { includeCaches: <см. этап 0> })`.
  3. `url = ${location.origin}${location.pathname}#p=${link}`.
  4. `navigator.clipboard.writeText(url)` + тоаст «Link copied».
- Если итоговая длина >`SHARE_SOFT_LIMIT_CHARS` (например, 6000) —
  модалка с двумя вариантами: «Share full (long URL)» / «Share without
  exact randomization caches» (тогда `includeCaches=false` и encode заново).
- Fallback на `document.execCommand('copy')` или модалку с строкой,
  если `navigator.clipboard` недоступен (например, http без https).

### Этап 3. Чтение ссылки на старте

Файл: `js/main.js`, в `_finishDesktopInit()` сразу после `initPresets()`.

1. `const hash = location.hash`. Если совпадает с `#p=...`:
2. `blob = decode(payload)`. На ошибке — тихо логируем, продолжаем
   как обычно.
3. Применяем blob — самый чистый путь:
   - временно сохраняем под зарезервированным именем `__shared__`
     (или `Shared · <name>`) в `presetManager.presets` **в памяти**, но
     **не пишем `savePresets()`**.
   - вызываем существующий `loadPreset('__shared__')`, который сделает
     всю работу с миграциями, кэшами и историей.
   - подменяем UI-надпись на `Shared · <name>` и помечаем
     `app.currentPresetName = '__shared__'`.
4. `history.replaceState(null, '', location.pathname)` — убираем hash из
   адресной строки, чтобы повторный share отдавал чистый URL.
5. Показываем баннер/плашку с предложением «Save to your library /
   Discard» (см. этап 4).

### Этап 4. «Save shared preset to my library»

- Плашка над dropdown’ом или в правом верхнем углу: «Open shared preset
  by friend — Save».
- На «Save» — переиспользуем `saveCurrentPreset()` (в `New` ветке, чтобы
  показать `promptPresetName`). Имя по умолчанию — берётся из shared blob.
- На «Discard» — `loadPreset('New')`, баннер скрывается.
- Если у получателя уже есть пресет с тем же именем — `confirmSaveOrNew`
  и/или предложение `Save as copy` (логика похожа на текущую при
  переименовании).

### Этап 5. Полировка

- **Тосты** через тот же сервис, что используется в проекте сейчас
  (если нет — мини-компонент с auto-dismiss 2.5 с).
- **Тесты** на decode стабильность:
  - Берём текущую ссылку из тестового набора.
  - Меняем дефолты в коде → прогоняем decode → проверяем, что
    результирующий blob равен ожидаемому (миграции отработали).
- **Документация:** короткий блок в `README.md` — «Sharing your preset».
- **Telemetry в dev-режиме:** `console.debug` с длиной payload и URL,
  чтобы быстро ловить регрессии при добавлении полей в `settings.values`.

## 6. Чего точно не делать

- ❌ Бэкенд, kv-сторадж, gist-как-хранилище. Лишняя зависимость без
  ощутимой пользы для нашего масштаба.
- ❌ Подпись / шифрование. Пресет публичный по сути.
- ❌ Передача `seeded`. Анимация Random-панели — только для встроенных
  seed-пресетов из `presets/`.
- ❌ Сохранение в `localStorage` без явного действия пользователя.
- ❌ Помещение payload в query-string (`?p=…`) — server-логи, длина,
  лишние перезагрузки.

1. **Дефолт по кэшам.** Включаем `includeCaches=true` всегда или только

   1024 символа в shared payload), или просто предупреждать?
3. **Поведение при импорте конфликта имён.** Auto-rename `Bushy (1)` или
   диалог `Replace / Save as copy / Discard`?
4. **QR-share.** Хотим ли вторую кнопку «Скачать QR» рядом с copy-link?
   Если да — её бюджет ~1.5 K, иначе деградировать в «слишком большой
   пресет для QR».
5. **Аналитика.** Считать ли что-то про share (счётчик, hash-ID)? Сейчас
   аналитики в проекте нет, лучше не вводить.

## 9. Журнал замеров (заполняется на этапе 0)

> Шаблон: `<имя пресета> — raw JSON: NN B / minified: NN B / deflate+base64url: NN B`

- _Measurements (seed `presets/*.json` raw file on disk ≈ exported JSON shape; deflate-raw matches share codec roughly):_

| preset file | JSON chars | deflate-raw bytes | ~`v1.`+base64url token chars |
| --- | ---: | ---: | ---: |
| tricky.json | 2262 | 659 | ~882 |
| anyway.json | 3586 | 846 | ~1131 |
| max-chaos.json | 3289 | 991 | ~1325 |
| bushy.json | 30954 | 4039 | ~5389 |
| nikitas-tiles.json | 57774 | 9233 | ~12314 |
| ellipsis.json | 39724 | 1480 | ~1977 |

- **Note:** share links use diff-vs-`pristineSettingsDefaults`, so URLs are typically much shorter than seed file sizes above.
- **Decision:** Adaptive `includeCaches` with modal near `SHARE_SOFT_LIMIT_CHARS` (6000 full URL chars) retained.

## 10. Якоря для контекста

- Текущая команда «выгрузить пресет JSON для папки `presets/`» —
  `VoidTypeface.exportSettingsPresetJson()` в `js/main.js` (toggle клавишей
  **J**, кнопка `#exportJsonBtn`). По форме её payload очень близок к
  тому, что нужно для share — отличие в diff-vs-default и компрессии.
- Анимация Random-панели на seed-пресетах — `pulseRandomAttention()` в
  `js/controllers/PresetsController.js`. **Не должна** срабатывать на
  shared-пресетах (через `shouldPulseRandomTutorialOnLoad` — проверяет
  `preset.seeded === true`).
- Список встроенных пресетов — `presets/index.json`. Их `seeded: true`
  выставляется только в `PresetManager.loadSeedPresets()`. `collectPresetData`
  при сохранении нового пресета **не** копирует флаг `seeded` (см. опцию
  `propagateSeeded`).

## 11. Как реанимировать этот план через год

1. Прочитать раздел 1 — подтвердить, что фича всё ещё актуальна.
2. Прочитать разделы 2–4 — проверить, не разъехалась ли архитектура
   (особенно `EXTRA_PRESET_SNAPSHOT_KEYS`, `pristineDefaults`,
   `loadPreset` миграции).
3. Раздел 7 — обсудить с собой текущие ответы (включаем кэши, лимит
   текста и т.д.).
4. Этапы 0 → 1 → 2 → 3 → 4 → 5 строго по порядку. Этап 0 — обязателен:
   браузерные лимиты и фактические размеры пресетов могли поменяться.
5. Не забыть актуализировать раздел 8: возможно, какие-то файлы
   переехали (мы любим рефакторить контроллеры).
